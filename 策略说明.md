## 模块化设计

策略代码独立在 `strategies/` 目录下，每个策略一个文件，通过统一的 `IStrategy` 接口与主服务交互。

### 目录结构

```
strategies/
  types.ts      — 共享类型和 IStrategy 接口定义
  registry.ts   — 策略注册表，管理所有策略实例
  s1.ts         — 策略1 · 常规加强
  s2.ts         — 策略2 · 常规
  s3.ts         — 策略3 · 扫尾
  s4.ts         — 策略4 · 反转
  s5.ts         — 策略5 · 概率追赶
```

### 策略接口

每个策略实现 `IStrategy` 接口：

- `checkEntry(ctx)` — 检查入场条件，返回方向或 null
- `checkExit(ctx, direction)` — 检查出场条件，返回止盈/止损信号和原因
- `updateGuards(ctx)` — 每个 tick 更新冷却锁等守卫状态
- `onEntryFilled(ctx, direction)` — 买入成交后回调，初始化追踪数据
- `resetState()` — 窗口切换时重置策略私有状态
- `getDescription()` — 返回前端 hover 描述（参数自动生成）
- `getStatePayload()` — 返回策略私有状态供前端展示

### 新增策略

1. 在 `strategies/` 下新建文件，实现 `IStrategy` 接口
2. 在 `types.ts` 的 `StrategyNumber`、`StrategyKey`、`ALL_STRATEGY_KEYS` 中加入新编号
3. 在 `registry.ts` 中 import 并注册
4. 重启服务，前端自动显示新策略的开关和描述

### 参数说明

每个策略文件顶部的常量即为可调参数，均带有注释说明。修改参数后重启服务即生效。前端 hover 描述会引用这些常量自动更新。

## 后端状态机

所有策略共用同一个状态机，同一时间只有一个策略持仓：

```
IDLE → SCANNING → BUYING → WAIT_FILL → HOLDING → SELLING → WAIT_SELL_FILL
                                                                    ↓
                                                        次数未满 → 回到 SCANNING
                                                        次数已满 → DONE
```

- 单局最大入场次数通过前端"单局 N 次"控件调整，持久化保存
- 窗口切换时全部重置

## 回测数据收集

前端"回测数据"按钮可开启数据收集，每秒记录一条 tick 数据到 `backtest-data/` 目录，用于离线分析和参数优化。
